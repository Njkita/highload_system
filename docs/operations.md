# Эксплуатация сервиса доставки еды

Документ описывает, как развёрнутый по [architecture.md](architecture.md) сервис живёт в проде: где физически крутится, как туда катится новый код, что мониторим и на что просыпаемся ночью, какой простой допустим по каждому домену, и сколько всё это стоит в Yandex Cloud сейчас, через двойной рост и через пятикратный. Числа по нагрузке, объёмам и SLA берутся из [requirements.md](requirements.md); архитектурные обоснования компонентов закрыты [ADR-001](adr/001-architecture-style.md)/[ADR-002](adr/002-data-storage.md)/[ADR-003](adr/003-payment-reliability.md), здесь они не пересказываются, только применяются.

## 1. Развёртывание

### 1.1. Что разворачиваем и под какую нагрузку

Под прод понимаем шесть прикладных сервисов из [architecture §2.2](architecture.md#22-c4-level-2--container-diagram) — `gateway`, `catalog`, `order`, `payment`, `tracking`, `notification` — плюс PostgreSQL, Redis, Kafka и Object Storage. Гипотезы по производительности уже проверены на PoC ([optimization-log.md](optimization-log.md)): инвариант `orders + outbox в одной транзакции` держится, idempotency-key, circuit breaker и cache-aside ведут себя как заложено. От PoC к проду меняются три вещи — managed-плоскость вместо self-hosted в docker-compose, multi-AZ резервирование вместо одного инстанса, и реальные объёмы данных вместо 500 ресторанов в seed.

Целевая рабочая точка берётся из [requirements §4](requirements.md#4-оценка-нагрузки-capacity-estimation) и закрепляется без округлений: 12 млн MAU, 2.4 млн DAU, расчётный пиковый онлайн ~104 тысячи, пиковый API-поток 2849 RPS (1781 на основном API, 971 на fallback-polling трекинга, 97 на статусные апдейты партнёров), 437 тысяч заказов в сутки, рост заказов 20% год к году, объём заказов и платежей с метаданными порядка 2.7 ТБ к концу первого года и около 10 ТБ к концу третьего. SLA — те же, что зафиксированы в [НФТ-003](requirements.md#нфт-003-availability): фиксация платежа 99.99%, создание заказа 99.95%, каталог и поиск 99.90%, уведомления 99.50%.

Профиль трафика остаётся read-heavy: примерно 80% запросов уходит в каталог, карточку ресторана, статус заказа и трекинг, оставшиеся 20% делятся между корзиной, созданием заказа, инициацией оплаты и обновлением статусов рестораном и курьером. Это влияет на топологию реплик и на выбор стратегии деплоя в [§2.2](#22-стратегии-деплоя-по-сервисам): чтение можно безопасно катить канареечно с разделением трафика, а write-критичный платёжный путь требует, чтобы новая версия успевала отработать webhook и reconciliation ещё до того, как её ставят под полную нагрузку.

### 1.2. Регион, зоны доступности и сетевая топология

Базовый регион — `ru-central1`. У него три independent зоны доступности (`ru-central1-a`, `ru-central1-b`, `ru-central1-d`), что покрывает требование по высокой доступности платёжного контура: при отказе одной зоны Patroni переключит писательскую роль на синхронную реплику, реплицированный Redis-шард не теряет хот-стейт, а брокер Kafka из оставшихся двух зон продолжает писать в топик `payments.events` при `min.insync.replicas=2`. Второй регион (`ru-central2`) держится холодным — для медиа и backup'ов, не для compute. Полностью geo-redundant active/active для платежей не делаем, потому что `synchronous_commit=remote_apply` ([ADR-002](adr/002-data-storage.md)) при инкрементальной cross-region латентности 30–60 мс начнёт убивать write-p99 на checkout'е, а пересмотр консистентной модели на quorum-репликацию — это отдельный проект, не вписывающийся в текущий бюджет ([requirements §5.2](requirements.md#52-ограничения)).

VPC организована классически: одна сеть, три подсети по `/20` (это 4094 адреса на зону, чего хватает на любые разумные сценарии масштабирования) — `10.10.0.0/20` в зоне a, `10.20.0.0/20` в b и `10.30.0.0/20` в d. В каждой подсети поднимается рабочий пул Managed Kubernetes; managed-сервисы (PostgreSQL, Redis, Kafka) — это собственные подсети в сервисной плоскости YC, к ним пробит security group routing из всех трёх рабочих подсетей. Egress в интернет идёт через NAT Gateway: с одной стороны, это требование PSP, который выдаёт IP-вайтлист только под фиксированный исходящий адрес; с другой — это даёт нам единое место для мониторинга outbound-трафика и его лимитов. CDN-домен `cdn.fooddelivery.ru` поднят на Yandex Cloud CDN с origin'ом в Object Storage и шилдингом в первой зоне.

DNS-зона `fooddelivery.ru` живёт в Cloud DNS с health-aware A/AAAA-записями на `api.fooddelivery.ru`: запись указывает на Application Load Balancer (L7), который терминирует TLS, проверяет HMAC-подпись на пути `/api/v1/webhooks/psp` и раздаёт трафик в target group из трёх backend-групп — по одной на зону. Эфемерные адреса подов на диаграмме не светятся, потому что они меняются по каждому restart'у и засоряют топологическую картину; стабильные FQDN — это `api.fooddelivery.ru`, `cdn.fooddelivery.ru`, `pg-fooddelivery.mdb.cloud.yandex.net`, `redis-fooddelivery.mdb.cloud.yandex.net`, `kafka-fooddelivery.mdb.cloud.yandex.net`. Service-to-service обращения внутри K8s идут по `*.fooddelivery.svc.cluster.local:<port>` через CoreDNS, что снимает необходимость хардкодить адреса соседей в каждом сервисе.

## 2. Стратегия деплоя

### 2.1. CI/CD pipeline

Pipeline разделён на четыре стадии в духе 12-Factor: `build → test → release → deploy`. Каждая стадия имеет свой артефакт, и один и тот же образ катится через dev → staging → prod, тегаясь по git sha и по семверу. Конфиг — GitHub Actions, `.github/workflows/*.yml` рядом с кодом; runner — GitHub-hosted на ubuntu-latest, для тяжёлых интеграционных тестов и нагрузки — self-hosted runner в Yandex Cloud, чтобы не платить за исходящий трафик из GitHub в YC.

На каждый pull request запускается build (Go 1.22, мультистейдж Dockerfile, `--platform=linux/amd64`), статический анализ (`golangci-lint`, `gosec`), юнит-тесты и интеграционные с эфемерным Postgres в контейнере. На merge в `main` — release-стадия: образ тегается `vX.Y.Z` и `<sha>`, пушится в Yandex Container Registry в зашифрованный репозиторий, проходит сканирование уязвимостей встроенным сканером YCR. Deploy-стадия отделена и не запускается автоматически: triggers — `workflow_dispatch` с выбором окружения и стратегии, плюс автоматический `staging` rollout по тегу `staging-*`. Прод поднимается отдельной командой через `kubectl apply` на манифесты, сгенерированные `helm template`, либо через Argo CD, если рабочая группа выбирает GitOps-модель; в обоих случаях источник истины — git, и любой деплой воспроизводим из тега.

Манифесты держим как код: либо Helm-чарт со values per environment, либо Terraform-модуль с CRD ArgoCD Application. Перекатить инфру вручную через UI запрещено — `terraform plan` обязателен перед любым изменением managed-сервисов или сетевой топологии, и review проходит через тот же pull request, что и код. Это даёт нам DR-сценарий, в котором `terraform apply` за 20–30 минут восстанавливает прод-окружение в новом регионе, не считая времени на восстановление БД из бэкапов.

### 2.2. Стратегии деплоя по сервисам

Унифицировать стратегию деплоя по всем шести сервисам было бы соблазнительно с точки зрения операционной простоты, но катастрофически плохо по riska/profit-балансу: профили у них разные, и одинаковое отношение к рисковому платёжному пути и к notification-консьюмеру неоправданно.

**Gateway (Envoy/nginx) едет Rolling**, потому что это stateless reverse-proxy без хранимого состояния и обратно совместимый между минорными релизами Envoy. `maxSurge: 25%, maxUnavailable: 0` — даёт прирост capacity на время раскатки и не теряет ни одного pod'а; за минуту-полторы пять подов из шести по зонам обновляются, рестарт каждого занимает порядка пяти секунд. Канарка для gateway избыточна: его поверхность узкая (роутинг + JWT + HMAC), и риск пробить какой-то edge-кейс закрывается интеграционными тестами на стадии release.

**Catalog едет Canary**, потому что это самый горячий read-сервис, и любая регрессия в SQL-плане или сериализации мгновенно ловится через p99 на главной странице. Поток разделяет ALB по weight'у backend-групп: 5% → 25% → 50% → 100%, переходы автоматические по metric gate в Argo Rollouts (или Flagger), gate'ом служит p99 search-операции и error rate (порог: p99 не вырос больше чем на 30% от baseline, error rate не превысил 0.5% — окно 3 минуты на каждой ступени). Полный путь раскатки занимает 12–15 минут; если на любой ступени gate проваливается, происходит автоматический rollback к предыдущей версии и алерт в дежурный канал.

**Order едет Rolling, но строго после Expand-миграций** ([§2.4](#24-миграции-базы-данных-по-схеме-expandcontract)). Контракт обратно-совместим между N и N+1 версией: новая версия может писать дополнительные колонки, но старая должна продолжать читать без ошибок. На время раскатки в БД одновременно работают v1 и v2 — для outbox это безразлично, потому что схема событий тоже расширяется аддитивно; для idempotency-key это безопасно, потому что Redis-кэш ответов нечувствителен к версии сервиса.

**Payment едет Canary с расширенными gate'ами**. Помимо p99 и error rate (как у catalog), gate'ит ещё и outbox lag (`max(now() - created_at) FROM outbox_events WHERE published_at IS NULL`): если относительно baseline он вырастет хотя бы на 30 секунд за 5-минутное окно — это признак, что новая версия неправильно пишет в outbox или подвисла на транзакции с `FOR UPDATE`. Доля канарки идёт ниже, чем у catalog (1% → 10% → 50% → 100%), потому что попадание бага в платёжный путь сразу ломает [НФТ-006](requirements.md#нфт-006-consistency-для-критичного-контура-заказа-и-оплаты) — двойные списания возникают именно на стыках. Полный rollout — 25–30 минут, что специально дольше catalog'а: на 10%-ступени мы проходим хотя бы один полный цикл webhook-reconciliation от PSP по реальным заказам.

**Tracking едет Rolling с длинным graceful drain'ом**. У него WebSocket-соединения держатся минутами, и просто закрыть их в SIGTERM'е недопустимо — клиент увидит обрыв трекинга курьера. Поэтому `terminationGracePeriodSeconds: 90`, на SIGTERM сервис сначала перестаёт принимать новые соединения через `/ready → 503`, отправляет существующим клиентам сигнал переподключения и держит до минуты на dрайн, после чего exit; клиент за это время поднимет новое соединение к другому экземпляру через sticky-сессию на ALB.

**Notification едет Rolling и допускает короткий простой**. Это consumer-only сервис без user-facing трафика; его SLA — 99.50% ([НФТ-003](requirements.md#нфт-003-availability)) — допускает до 3 часов 39 минут простоя в месяц, что позволяет даже Recreate-стратегию в окно низкой нагрузки (3:00–5:00 MSK). Канарка тут — оверкилл: либо новая версия консьюмерит топик, либо нет, а на боевой нагрузке достаточно секунд, чтобы увидеть рост lag'а через `kafka_consumergroup_lag`.

### 2.3. Zero-downtime контракт: probes и graceful shutdown

Без явного контракта между приложением и оркестратором Rolling и Canary превращаются в «накатили v2 — теряем in-flight запросы пользователей». Контракт держится на трёх probes и пяти шагах graceful shutdown'а.

`/healthz` (liveness): отдаёт `200 OK` если процесс жив и event loop не залип; не пытается дёргать БД или Redis, потому что транзитный отказ зависимостей не должен убивать pod — это лечится в circuit breaker'е, не в рестарте. `/readyz` (readiness): отдаёт `200 OK` если процесс готов принимать трафик — pgxpool инициализирован и есть хотя бы один свободный коннект, Redis-пинг ответил за 50 мс, для consumer'ов добавляется проверка, что Kafka client подключён и offsets fetched. `/startupz` (startup): отдаёт `200 OK` после полной инициализации — миграций, прогрева кэшей, регистрации в service mesh; до этого момента liveness и readiness отвечают `503`, что не даёт K8s рестартануть pod, который ещё штатно стартует.

Graceful shutdown идёт по сценарию: K8s шлёт SIGTERM и удаляет pod из endpoints Service, после этого приложение разворачивает readiness на `503` (это запускает event-driven отключение upstream'а в Envoy за 1–2 секунды), перестаёт принимать новые HTTP-запросы и Kafka-сообщения, дослуживает все in-flight в пределах `terminationGracePeriodSeconds`, закрывает соединения с PostgreSQL и Redis (важно: dialer должен ждать flush WAL для outbox-записей, иначе можно потерять только что закоммиченное событие на стороне приложения, хотя в БД оно уже есть), закрывает Kafka consumer'а с commit'ом offset'ов на последнем успешно обработанном сообщении (`at-least-once` — допускаем повторную обработку, она безопасна благодаря идемпотентным consumer'ам), и только потом возвращает `exit(0)`. Если за `terminationGracePeriodSeconds` приложение не успело — K8s шлёт SIGKILL; такой кейс должен попадать в алерт и в постмортем.

Конкретные таймауты подобраны под профили: gateway — 10 секунд, потому что upstream-call'ы укладываются в 2.5 секунды; catalog/order — 30 секунд; payment — 30 секунд при сохранении возможности дослужить webhook PSP в идемпотентной транзакции; tracking — 90 секунд под drain WebSocket'ов; notification — 60 секунд под доедание прочитанного batch'а из Kafka. Все эти значения зашиты в Helm-чарте на уровне Deployment-манифеста, не в коде сервиса — иначе SRE не сможет подкрутить под изменившийся профиль трафика без релиза.

### 2.4. Миграции базы данных по схеме Expand/Contract

Главная боль highload-деплоя — что при Rolling и Canary v1 и v2 одновременно работают с одной и той же БД, и любая ломающая миграция (drop column, rename, type change) внутри одного релиза мгновенно даёт `500`-ки на половине трафика. Поэтому единственный допустимый процесс — Expand/Contract: каждая миграция аддитивна и обратно совместима с предыдущей версией кода.

Возьмём для примера переименование `delivery_address.street_line` в `delivery_address.address_line` — заказчик сказал, что в части регионов СНГ адресная схема не делится на street/house. Релиз N (expand) добавляет `address_line TEXT NULL` рядом со старым полем: миграция не блокирует таблицу, потому что в PostgreSQL 11+ добавление nullable-колонки не переписывает heap. Релиз N+1 (deploy v2) — v2 умеет писать в обе колонки и читать `COALESCE(address_line, street_line)`, v1 продолжает работать со старым полем, ничего не ломая. Релиз N+2 (backfill) запускает фоновый воркер, который пакетно (по тысяче строк за раз, с `pg_sleep(0.05)` между батчами) переносит данные из `street_line` в `address_line`, после завершения переключает чтения только на новую колонку. Релиз N+3 (contract) — `DROP COLUMN street_line`, только после того как телеметрия показала, что никто из живых сервисов больше не пишет в старую колонку.

Это медленно — четыре релиза вместо одного, в зависимости от частоты раскаток это занимает от двух дней до двух недель. Но альтернативы нет: drop column в одном релизе со сменой кода либо требует maintenance window с полной остановкой (что несовместимо с 99.95% SLA на создание заказа), либо ломает trafic во время Canary. Тот же шаблон применяется для смены типа колонки (через временную колонку нового типа), переименований таблиц, расщепления одной таблицы на две. Migration tool — `golang-migrate` с `up`/`down` файлами в репозитории, hash-pinned версии, integrity hash проверяется при старте сервиса.

Никаких автоматических rollback-миграций после прохождения через прод не делаем: если v2 упал и мы катимся обратно к v1 — миграции остаются на месте, потому что они expand-only. Эта политика — следствие [feedback'а от пользователя](../README.md): rollback требует явного согласия, и даже при регрессии не катим назад без подтверждения.

## 3. Observability

### 3.1. Четыре алерта на critical path

Бюджет — четыре алерта, по Golden Signals, на самом критичном бизнес-пути системы. Critical path определяется не «где больше всего трафика», а «где боль клиента наибольшая и потеря денег прямая», и поэтому это не каталог, а платёжный контур: оплата заказа от инициации до подтверждения от PSP. Все четыре алерта — на симптомы, наблюдаемые пользователем или прямым business-effect'ом, не на ресурсы (CPU/RAM остаются в дашбордах, не в paging-канале).

| Сигнал | Метрика | Порог + окно | На что |
|---|---|---|---|
| **Latency** | `histogram_quantile(0.99, http_request_duration_seconds_bucket{path="/api/v1/orders/{id}/payment", method="POST", status=~"2.."})` — только успешные | > 1500 мс 5 минут подряд | Payment Service: либо PSP тормозит за пределами latency-budget из [requirements §4.4](requirements.md#44-latency-budget-для-критичного-сценария), либо мы сами уплыли на этапе записи в БД до вызова PSP |
| **Errors** | `sum(rate(http_requests_total{path=~"/api/v1/orders.*", status=~"5.."}[5m])) / sum(rate(http_requests_total{path=~"/api/v1/orders.*"}[5m]))` | > 0.5% 5 минут подряд | Order Service: либо БД-проблемы (transaction abort на outbox), либо сломалась идемпотентность, либо упёрся в pgxpool — все варианты ломают [НФТ-006](requirements.md#нфт-006-consistency-для-критичного-контура-заказа-и-оплаты) |
| **Traffic** | `sum(rate(kafka_outbox_published_total[5m]))` — публикации из outbox relay в Kafka | падение ниже 50% от EWMA-7d-baseline 10 минут подряд | Outbox relay: pod упал, advisory-lock не передан, или БД в overload и polling не успевает. Внешне это невидимо до момента, когда заказы начнут уходить без оплаты или клиент не получит push о подтверждении |
| **Saturation** | `pg_stat_activity_count{state="active"} / pg_settings_max_connections` либо outbox lag `max(now() - created_at) FROM outbox_events WHERE published_at IS NULL` либо backlog age у payment-consumer'а `time() - kafka_consumergroup_oldest_unconsumed_timestamp{group="payment-worker"}` | первая метрика > 75% 5 минут, вторая > 60 секунд 3 минуты, третья > 120 секунд 5 минут | PostgreSQL, outbox relay и Kafka-consumer: connection-pool насыщен, выстраивается очередь, или сообщения зависли. Это упреждающий алерт — пока ещё не пробит SLA, но скоро будет |

Третий угол насыщения — это backlog **по возрасту**, а не по offset'у. Классический `kafka_consumergroup_lag` в сообщениях обманчив: при равномерном producer-throughput'е лаг в 50 000 сообщений может ничего не означать (просто batch processing), а при 100 сообщениях, но возрастом в 10 минут, реально означает зависший partition или умершего consumer'а, которого rebalance не подхватил. Age-based метрика ловит этот класс инцидентов на минуты раньше offset-метрики.

Latency считается только по успешным запросам (`status=~"2.."`), потому что иначе быстрые `503` от circuit breaker'а будут маскировать медленные успехи, и метрика станет лживой. Все четыре алерта routed в PagerDuty / Telegram-бот SRE-rotation, severity `high`, окно тишины (silence) автоматически выставляется на 30 минут после деплоя — это даёт раскатке отработать без false-positive'ов на короткий выброс latency во время прогрева. CPU, memory, дисковая нагрузка, длина очереди Redis, hit-ratio кэша, RPS на разных эндпоинтах и десятки других метрик остаются в дашбордах: они нужны во время разбора инцидента, но не должны будить дежурного.

Логика «алерты на симптомы, не на ресурсы» — это правило Google SRE, и оно держится за тем, что CPU 95% в одной зоне сама по себе не значит ничего, если p99 пользовательских запросов не пострадал; и наоборот, CPU 30% не значит, что всё хорошо, если БД ушла в swap и latency взлетела. Симптом — это то, что чувствует клиент или бизнес; ресурс — это диагностический контекст. Перепутаешь их — будешь будить SRE на хайповые но безобидные пики, и реальные деградации пройдут мимо.

### 3.2. Дашборды на трёх уровнях

Один дашборд на пятьдесят графиков читать невозможно: глаз не цепляется, графики смешиваются, важные сигналы тонут. Поэтому дашборды организованы в трёхуровневую иерархию, где сверху видно «что сломалось», а спуск даёт «почему».

**Уровень 1 — Overview.** Один экран на всю систему, Golden Signals по top-5 пользовательских операций: `GET /restaurants`, `GET /restaurants/{id}`, `POST /orders`, `POST /orders/{id}/payment`, `GET /orders/{id}`. На каждой строке — четыре графика: latency p50/p99, error rate, RPS, насыщение (queue depth или pool used) для соответствующего сервиса. Дополнительно — четыре «жирные» цифры в шапке: текущий error budget burn rate для платёжного домена (на 99.99% SLA это `0.01% / 30d`, и burn rate > 1.0 означает, что мы за 30 дней потратим больше бюджета на ошибки, чем имеем), доля заказов в состоянии `payment_pending_confirmation` старше 5 минут, средний возраст самого старого неопубликованного outbox-события, и пятиминутный RPS на платёжных эндпоинтах. Этот экран SRE открывает первым при любом инциденте и через 10 секунд видит, где проблема — у клиента, у БД или в outbox.

**Уровень 2 — Service-level.** По одному дашборду на каждый из шести сервисов, плюс отдельный для outbox relay и reconciliation worker'а. На каждом — полный RED-стек: rate по эндпоинтам и методам, error rate с разбивкой по кодам (4xx и 5xx отдельно — это разные миры, 4xx обычно говорит о клиенте, 5xx о нас), latency p50/p95/p99/p99.9 по эндпоинтам. Здесь же — бизнес-метрики, специфичные для сервиса: для catalog'а это hit-ratio Redis-кэша и распределение размеров результатов поиска; для order'а — среднее количество позиций в заказе и доля идемпотентных повторов; для payment'а — доля circuit breaker OPEN-событий и распределение latency PSP'и; для notification'а — длина очереди в Kafka и доля DLQ-сообщений. Cюда заходит SRE после Overview, чтобы понять, какой именно сервис страдает.

**Уровень 3 — Diagnostic.** USE-метрики на ресурсы (CPU, RAM, disk I/O, network) per pod, per node, per managed-instance; traces для конкретного запроса через Jaeger или встроенный в Yandex Cloud OpenTelemetry-агент; pg_stat_activity для PostgreSQL с детализацией по самым медленным запросам и блокировкам; `INFO replication`/`SLOWLOG GET` для Redis. Этот уровень не имеет фиксированной формы — он строится под расследование: SRE вводит trace_id, найденный по логам или exemplar'у на Overview, и проваливается в waterfall span'ов, доходя до конкретного span'а в payment-сервисе или конкретного SQL в БД. Связка — через trace_id в обязательных полях лога ([§3.3](#33-логи-формат-обязательные-поля-политика-pii)) и через exemplars в Prometheus-метриках.

Принцип такой: метрика на overview отвечает на «что-то сломалось», лог на service-level — «у какого сервиса и в какой момент», трейс на diagnostic — «где конкретно внутри запроса». Три столпа observability — это не три отдельных мира, а три взгляда на одни и те же события, связанные через trace_id.

### 3.3. Логи: формат, обязательные поля, политика PII

Формат — structured JSON в stdout. Это позволяет Cloud Logging (managed-сервис YC) собирать логи без отдельного агента, парсить поля для индексации и поиска, не теряя структуру, и хранить эффективно. Альтернатива — текстовые строки в `syslog`-формате — отвергнута, потому что в распределённом окружении на 30+ pod'ах построить корреляцию по тексту через grep — задача на час; по структурированному JSON через Cloud Logging Query Language — задача на минуту.

Обязательные поля в каждой записи: `ts` (UTC, RFC 3339 с микросекундами), `level` (debug/info/warn/error), `service` (имя сервиса), `version` (тег образа), `env` (prod/staging/dev), `trace_id` (W3C TraceContext, прокидывается через все вызовы, генерируется в gateway если клиент не прислал), `span_id` (текущий span внутри запроса), `req_id` (X-Request-Id, прокидывается через все upstream-вызовы), `user_id` (захэшированный SHA-256 от реального user_id с пайпер-солью, никогда в открытом виде), `msg` (короткая человеко-читаемая строка), `attrs` (произвольные ключи под кейс — order_id, payment_id, psp_payment_id, error_code и так далее). При ошибке добавляется `stack` со stack trace и `cause` с цепочкой wrap'нутых ошибок.

Логируем: каждый входящий HTTP-запрос (на старте — `event=request.start` с методом, путём, request size; на завершении — `event=request.end` с status, response size, duration_ms); каждый исходящий call к внешним системам (PSP, антифрод, push-gateway) — start и end, с явным `target=psp.yookassa` и `intent_id` чтобы reconciliation мог восстановить контекст; любая ошибка — `level=error` с полным stack'ом и cause-chain'ом; любой бизнес-критичный переход — создание заказа, успешный платёж, отмена с компенсацией refund'а, попадание в DLQ.

Не логируем: полное имя клиента (только хэш user_id'а), полный адрес доставки (только city + первые буквы улицы для отладочной видимости, без номера дома и квартиры), номер карты (никогда — PSP даёт нам только masked last4, его и пишем), полный JWT-токен (только jti), номер телефона (маскируем last4), email (хэш для корреляции, без localpart). Comment'ы клиента к заказу — частично: до 30 символов и без regex'ов на телефон/email; иначе клиент пишет своё мобильное в комментарии «позвонить если что» и оно уходит в open log'и.

Retention: warm-storage в Cloud Logging — 30 дней, после чего лог-партиции автоматически уезжают в Object Storage Cold (90 дней) и потом в Ice (год). Это даёт нам возможность дотянуться до года назад для аудита платежей и закрывает требование 152-ФЗ по хранению логов доступа к PII (по сути логи доступа к данным заказа — это access log, которые мы должны хранить 6 месяцев). Запросы клиента на удаление аккаунта (право на забвение) проходят через отдельный pipeline: помечаем user_id'хэш как `forgotten`, при последующих запросах из логов хэш не разрешается обратно в идентификатор.

## 4. Доступность

### 4.1. Целевые SLO по доменам

Бизнес-обоснование — где конкретно мы теряем деньги при простое, и сколько клиенты прощают. Главный фактор — это конверсия checkout'а: средний чек на сервисе доставки еды по российскому рынку колеблется в районе 1200–1500 рублей, заказов в час в обеденный пик — около 18 тысяч (`peak online ≈ 100k × 0.14 / 50min ≈ 18k/час`), так что час простоя checkout'а — это потенциально 25 миллионов рублей упущенной выручки. Час простоя каталога — менее болезненно, потому что часть пользователей возвращается через минуту, и заказы делаются всё равно (по нашей оценке возврат — около 60% за 15 минут, дальше exponential decay).

Из этого выводятся SLA по доменам, уже зафиксированные в [НФТ-003](requirements.md#нфт-003-availability):

| Домен | SLO | Допустимый простой в месяц | Бизнес-обоснование |
|---|---:|---:|---|
| Фиксация платежа | 99.99% | 4.4 мин | Прямая потеря денег и юридическая ответственность за расхождение состояния; PSP отдельно не страхуют нашу сторону |
| Создание заказа | 99.95% | 21.9 мин | Потеря заказа на checkout'е почти не восстанавливается — клиент уходит к конкуренту |
| Корзина | 99.95% | 21.9 мин | Отказ корзины — это «не могу заказать», восстановление частично через локальный кэш клиента |
| Каталог и поиск | 99.90% | 43.8 мин | Лёгкая деградация: можно подменить cached-ответом, клиенты возвращаются |
| Трекинг заказа | 99.90% | 43.8 мин | Заказ уже сделан; в худшем случае клиент звонит в поддержку и узнаёт статус голосом |
| Уведомления | 99.50% | 3 ч 39 мин | Best-effort; SMS/push — это удобство, не критичный канал, fallback — клиент сам открывает приложение |

SLO следит дежурный SRE через дашборд error budget'а на Overview ([§3.2](#32-дашборды-на-трёх-уровнях)). Перерасход бюджета за 30-дневное окно автоматически включает feature freeze в платёжном домене: до момента, пока burn rate не вернётся к норме, релизятся только bug-fix'ы, новые фичи откладываются. Это политика, явно записанная в runbook и принятая product-командой; без неё SLO остаётся декларацией, к которой не привязано ничего материального.

### 4.2. RPO и RTO по компонентам

RPO (Recovery Point Objective) — сколько данных мы готовы потерять при катастрофе, измеряется во времени между моментом катастрофы и моментом последнего восстанавливаемого backup'а. RTO (Recovery Time Objective) — за сколько мы готовы поднять систему обратно. Они тянут стек в противоположные стороны: RPO = 0 требует синхронной репликации (дорого по latency), RTO ≤ 15 мин — автоматического failover и заранее прогретого standby (дорого по железу). Каждый компонент закрывается отдельно, потому что у платежей и у кэша совершенно разные требования.

**PostgreSQL для платежей и заказов (`orders_db`).** `RPO = 0`, `RTO ≤ 15 минут`. Достигается тройкой: synchronous replica в соседней зоне (Patroni + `synchronous_commit=remote_apply`, на write-критичные таблицы — `payments`, `orders`, `outbox_events`), асинхронная реплика в третьей зоне как кандидат на promote при двойном отказе, и непрерывный stream WAL'а через WAL-G в Object Storage с PITR-возможностью до 7 дней назад. Failover автоматический через Patroni при потере primary — promote синхронной реплики занимает 10–30 секунд; восстановление из бэкапа в новом регионе при тотальной катастрофе — около часа на pg_basebackup + WAL replay (это уже не RTO платежей, это DR-сценарий).

**PostgreSQL для каталога (`catalog_db`).** `RPO ≤ 15 минут`, `RTO ≤ 1 час`. У каталога нет инварианта по money flow, потери последних апдейтов меню допустимы (ресторан их перевыставит). Async-replica в соседней зоне, snapshot'ы каждые 6 часов, восстановление из snapshot'а в случае отказа. `synchronous_commit=local` экономит write-latency, что важно при batch-обновлениях меню от ресторанов.

**Redis Cluster.** Состояние делится на пять классов с разными требованиями. **Корзины** — `RPO ≤ 1 минута`: AOF `everysec` + cluster-replica в соседней зоне; потеря десятка-другого секунд изменений корзины это не критично, клиент перевыставит. **Idempotency-ключи** — `RPO ≤ 5 минут`: даже их потеря не катастрофична, в БД есть UNIQUE-constraint на `idempotency_key`, и повторный запрос либо вернёт тот же result через DB lookup, либо честно создаст новый заказ (что согласуется с UX). **Hot-cache карточек ресторанов и поиска** — `RPO not applicable`: cache rebuild'ится сам, при потере хитимся в БД, p99 каталога временно проседает с 1 мс до 100 мс, что несравнимо лучше любых синхронных гарантий. **Hot-state заказа (`order_hot:{id}`)** — `RPO ≤ 1 минута`: восстанавливается из событий `orders.events` в Kafka через replay. **GEO курьеров** — `RPO not applicable`: курьер шлёт следующую координату через 10 секунд, потеря последней позиции для пользователя означает «карта чуть устарела», что приемлемо.

**Managed Kafka.** `RPO = 0` для критичных топиков (`orders.events`, `payments.events`), `RTO ≤ 5 минут`. Достигается тройкой брокеров с `acks=all`, `min.insync.replicas=2`, replication factor 3 — потеря одного брокера или одной зоны не теряет ни одного подтверждённого сообщения. Retention 14 дней покрывает любой реалистичный сценарий восстановления consumer'а после крупного инцидента, а replay через `kafka-console-consumer --from-beginning` для критичной аналитической пересчётки — стандартная операция.

**Object Storage.** `RPO ≈ 0`, `RTO ≤ 1 час`. YC даёт встроенную репликацию внутри региона на три копии, cross-region replication в `ru-central2` мы включаем для bucket'а `fooddelivery-photos` (медиа) и `fooddelivery-backups` (WAL-G); версионирование и lifecycle на основной бакет — стандартные 90 дней Standard → 365 Cold → archive Ice.

**Stateless-сервисы.** `RPO not applicable`, `RTO ≤ 1 минута`. State либо в Redis (ephemeral), либо в БД (durable); при отказе пода другой инстанс подхватывает трафик через service discovery за десятки секунд.

### 4.3. Резервирование: active/active vs active/standby

Решение по каждому компоненту следует из его профиля по записи и из готовности базовой технологии к multi-master.

Stateless-сервисы — **active/active по всем трём зонам**, на K8s через `podAntiAffinity` (две реплики одного сервиса не оказываются в одной зоне). ALB сам по себе работает active/active на edge'е, распределяя трафик между target group'ами разных зон. Redis Cluster — **active/active**: каждый шард имеет master в одной зоне и реплику в другой, ключи шардятся по hash slot'ам, чтение можно направить на replica для разгрузки master'а. Kafka — **active/active**: брокеры распределены по трём зонам, лидер партиции живёт в одной зоне, ISR-реплики в двух других.

PostgreSQL — **active/passive**: один primary, одна синхронная и одна асинхронная реплика. Multi-master в Postgres (через BDR или Citus с distributed transactions) сильно сложнее в эксплуатации и не оправдан при нашем профиле write-нагрузки (48 RPS критичной записи). Patroni обеспечивает автоматический failover за 10–30 секунд, что укладывается в RTO платежей. Object Storage — **active с холодным DR-реплицированием**: основной бакет в `ru-central1` активный, реплика в `ru-central2` только под чтение, переключение CDN-origin'а на DR-бакет — ручная операция в DR-сценарии (это не отказ зоны — это отказ региона, события маловероятного, но фатального).

Гео-распределённость, выходящая за пределы одного региона, для compute и БД сознательно не делается. Cross-region active/active требует переосмысления модели консистентности (либо eventual consistency на платежах, либо paxos/raft через регионы с latency-budget'ом, который мы не вписываем в [requirements §4.4](requirements.md#44-latency-budget-для-критичного-сценария)). Single-region многозональная топология даёт реальную доступность 99.95% и выше при правильной операционной дисциплине, что достаточно для бизнеса и сильно дешевле полноценной мультирегиональности.

### 4.4. Maintenance window

Регулярные деплои не требуют maintenance window: они идут Rolling или Canary, без даунтайма, с graceful shutdown'ом и Expand-only миграциями ([§2](#2-стратегия-деплоя)). Major-апгрейд PostgreSQL (раз в 1–2 года, например 16→17) тоже делается без окна — через blue-green: рядом с действующим `orders_db` поднимается параллельный кластер новой major-версии, через `pg_logical` (или встроенный механизм logical replication в YC Managed) идёт репликация всех изменений с старого primary на новый, catch-up занимает часы и работает в фоне. В момент готовности — короткое окно (десятки секунд) на quiesce write'ов через advisory-lock на уровне приложения, контроль `pg_current_wal_lsn()` обоих кластеров на совпадение, переключение `DATABASE_URL` в Helm-чарте и rolling-перезапуск сервисов. Старый кластер живёт ещё сутки на случай отката, потом удаляется. Это процедура на инженерный день, но окна простоя в ней нет.

Окно нужно только для двух классов операций, где без даунтайма не получается принципиально: major-апгрейд Managed Kubernetes control plane (раз в полгода, координируется с YC и проходит за 5–10 минут на каждой ноде по очереди) и физическое переселение между типами managed-инстансов с изменением storage-tier'а или сетевой схемы (раз в годы, если вообще).

Окно — ночь со вторника на среду или со среды на четверг, 03:00–05:00 MSK. Это эмпирически самое тихое время по нашему профилю: обеденный и вечерний пики уже отыграны, утренний кофе ещё не начался, восстановление при инциденте занимает один-два часа без необходимости поднимать руководство. Частота — не чаще одного раза в квартал, анонс — за две недели через статус-страницу, рассылку партнёрам (рестораны) и push-уведомление клиентам за сутки. В обеденный пик (11:00–14:00 MSK) и в вечерний пик (18:00–21:00) — никаких операций, включая non-critical deploy; в субботу-воскресенье (когда нет привычного «обеда в офисе», но есть продлённый домашний пик) — только бизнес-критичные hotfix'ы.

## 5. TCO в Yandex Cloud

Расчёт идёт по управляемым сервисам YC: для compute это Managed Kubernetes с собственными worker-пулами, для БД — Managed PostgreSQL, Managed Valkey (Redis-совместимый по протоколу managed-сервис YC, заменивший Managed Redis для новых проектов после смены лицензии Redis Labs в 2024 — со стороны приложения это тот же `go-redis/v9`, разница только в названии в биллинге), Managed Kafka; для медиа — Object Storage + Cloud CDN, для периметра — Application Load Balancer и NAT Gateway. Альтернатива — самостоятельно поднятые VM с self-hosted Patroni / Redis Sentinel / Kafka — отвергнута, потому что операционная нагрузка по бэкапам, failover, патчам и мониторингу managed-сервисов измеряется приблизительно в треть FTE на каждый кластер, и при бюджете курсового проекта это сжирает экономию по чистым vCPU. Цены — Россия, со включённым НДС, дата выгрузки — апрель-май 2026 по pricing.md соответствующих сервисов; часть позиций (storage ₽/GB·мес, операции Object Storage, egress ₽/GB) на сайте Yandex Cloud рендерится за SmartCaptcha и в исходном виде недоступна, по этим позициям проставлен порядок величины с пометкой «приблизительно».

Со CUD на 12 месяцев скидка на `vCPU + RAM` компонент достигает 40–49%; накладываясь на Compute и managed-instance'ы Postgres/Redis/Kafka, это даёт экономию порядка 130–150 тыс. рублей в месяц. Финальная цифра по инфраструктуре под текущую нагрузку с CUD — около **350–360 тысяч рублей в месяц**.

Большая статья — Cloud Logging. Если оставить ingest «как есть» (полные логи каждого запроса × 30+ pod'ов × 30 дней), получится 4–5 ТБ в день, что даёт около 500–700 тысяч рублей в месяц только на логи. Sampling на стороне сервиса (info-логи 20%, error 100%, debug никогда в проде) + аггрегация на gateway-уровне сокращает поток до 1.5 ТБ/мес. Альтернатива — собирать логи в собственный Loki/ELK-кластер на отдельной VM, что даёт экономию по тарифу, но добавляет операционной работы; на текущей фазе остаёмся на managed-варианте.

### 5.2. Что меняется при росте в 2x (24M MAU, ~6k RPS пик)

Compute масштабируется почти линейно, но не строго 2x: read-сервисы можно оставить на тех же узлах с +50% replicas через HPA, потому что Redis-кэш ([architecture §2.2](architecture.md#22-c4-level-2--container-diagram)) забирает большую часть нагрузки от БД на каталоге. Реально нужно: + worker-нод примерно на 60% (поверх существующих 9 + ещё 5–6), что даёт +₽55k/мес. Compute итого: ~₽150k/мес.

PostgreSQL по записи — главное место, где 2x болезненно. Pgxpool на `orders_db` упрётся в 128 max_connections, нужно либо разнести на два кластера (`orders_db` и отдельно `payments_db`), либо вертикально масштабировать ноды до `16 vCPU / 64 GB`. Простой и безопасный путь — вертикальный апскейл, который даёт ~+50% к биллингу: ~₽190k/мес. Async-реплику можно добавить под read-нагрузку аналитики, +30k.

Redis 2x: один из трёх шардов становится горячим, добавляем ещё один шард (всего 4 × master+replica), +₽15k. Kafka 2x: партиции увеличиваются до 24 на топик `orders.events`, +50% брокерской мощности, +₽25k.

ALB при 6k RPS: 12 RU × ₽2.63 × 730 = ₽23k базы, плюс пик-резерв, ~₽30k. Egress ×2: ~₽24k. Cloud Logging при 2x — это серьёзная статья роста: ingest вырастает с 1.5 до 3.5 ТБ/мес (нелинейно — увеличивается доля error'ов под нагрузкой, +retry-логи), что даёт ~₽220k. Object Storage растёт медленнее, +50% на медиа: ~₽13k.

**Итого 2x под нагрузкой: ≈ ₽800–850 тысяч в месяц**, с CUD — ≈ **570–620 тысяч**.

### 5.3. Что меняется при росте в 5x (60M MAU, ~15k RPS пик)

5x — это уже точка, где архитектура должна меняться, а не масштабироваться линейно. Несколько важных эффектов.

Catalog при 15k RPS на поиске уйдёт за p99 < 300 мс на `pg_trgm + PostGIS` в нашем PostgreSQL, и это активирует Day-2 миграцию на OpenSearch ([ADR-002](adr/002-data-storage.md)). OpenSearch-кластер в YC — это ещё ~₽120k/мес базы, плюс затраты на CDC-пайплайн через Debezium, плюс инженерное время на стабилизацию.

PostgreSQL по записи при ~120 RPS критичной нагрузки уже не справится вертикальным апскейлом одного primary: либо разделение на `payments_db` + `orders_db` физически, либо переход на Citus с шардированием по `user_id` (последнее переписывает наш SQL-слой). Простой путь — два HA-кластера + один read-replica общий: +₽250k.

Redis при 5x по корзинам и hot-cache требует 6 шардов с двойной репликацией. Доп инстансы +₽120k. Kafka 5x: 60 партиций на критичные топики, 6 брокеров с увеличенной storage'ью, +₽75k.

ALB и CDN масштабируются почти линейно. Логи при 5x ингесте — это уже 7–10 ТБ/мес, при которых managed Cloud Logging становится дороже самостоятельной альтернативы, и приходится строить свой кластер на одном из managed-вариантов (ClickHouse + Vector, либо OpenSearch для логов). Это разовая инженерная работа на месяц и +₽80k опер.

Egress 5x на API + CDN на 5 ТБ/мес медиа: ~₽60k. Object Storage растёт линейно, +120k.

Главное архитектурное изменение на 5x — это **выход за пределы одного региона**. При 60M MAU СНГ-география начинает иметь смысл (Беларусь, Казахстан, Узбекистан — разная сетевая локация), и появляется потребность в edge-узлах. Это либо CDN с edge-функциями для checkout'а, либо реальный multi-region deployment с переосмыслением консистентности платежей. На 5x ставим pin: этот вопрос отдельным проектом, не TCO-расчётом одной таблицы.

**Итого 5x с многократным архитектурным разворотом: ≈ ₽1.8–2.2 миллиона в месяц**, с CUD — ≈ **1.3–1.6 миллиона**.

Сравнение базы и 5x даёт сублинейный эффект: пятикратный рост нагрузки выливается в инфра-cost ×3.6–4.4 (в зависимости от того, считать с CUD или без). Источник амортизации — фиксированные компоненты, не растущие с RPS (Managed K8s control plane, минимальный ALB-резерв, базовые managed-инстансы Postgres/Valkey/Kafka), CUD-скидки на 12-месячный коммит и hit-rate Redis-кэша, который при горячем каталоге растёт быстрее RPS и снимает с БД основную часть прироста. Самые нелинейные позиции, тянущие соотношение вверх — Cloud Logging (× ~7 от базы из-за всплеска error- и retry-логов в overloaded-режиме) и PostgreSQL write-pool (требует разделения на два кластера или Citus-шардирования, и тут шаг получается ступенькой ×4–5, а не плавно). Сообщать заранее «5x нагрузки ≈ 5x cost» в product-планёрке — значит закладывать запас в полтора раза больше реального, поэтому в capacity-плане эта пропорция фиксируется явно.

### 5.4. Операционные затраты

Помимо инфраструктуры, нужны люди: SRE-rotation на on-call, security audit (раз в год независимая команда), внешние SaaS, не покрытые YC (Sentry для error tracking как опция, статус-страница типа Atlassian Statuspage), резервные доменные имена.

Базовая команда — 2 SRE с зарплатным фондом порядка ₽350–450k каждый по российскому рынку 2026 года = ₽700–900k FOT/мес. On-call rotation 24/7 у двоих закрывает базовый сценарий, при росте сервиса до 5x ставится третий. Это самая большая операционная статья, и она не масштабируется линейно от MAU: 2 SRE достаточно и на 12M, и на 24M, и до примерно 35–40M MAU. После — третий, и обязательно специализация (один больше про базы, один больше про edge/network).

Security audit — ~₽300–500k разово, раз в год. Sentry / Bugsnag — ~₽5–10k/мес если нужен. Status page — ~₽3k/мес. Прочая мелочёвка (домены, SSL, on-call PagerDuty если не используем Telegram-бота) — суммарно до ₽10k/мес.

Сумма операционных по году: ~₽12–15 миллионов, что в пересчёте на месяц — ~₽1–1.3 миллиона. Это **в три раза больше** инфраструктуры YC на текущей нагрузке, и в этом нет ошибки: для highload-системы стоимость людей всегда доминирует над стоимостью железа, пока MAU не вырастет в десятки и сотни миллионов.

### 5.5. Что дороже всего и почему

В порядке убывания:

**Cloud Logging** даже после агрессивного sampling'а — это 20–25% инфраструктурного TCO, и оно нелинейно растёт с нагрузкой. Причина — мы пишем структурированный JSON по каждому запросу и ошибке, под 30+ pod'ов, и хранение 30 дней быстро уходит в терабайты. Закрывается тремя приёмами: sampling info-уровня, retention'ом 7 дней warm и быстрым перекидыванием в Cold Object Storage, выносом части логирования в свой OpenSearch/ClickHouse при росте.

**Managed PostgreSQL для платежей** — 25–30% инфры. Цена расплаты за `synchronous_commit=remote_apply` + большие storage'и + high availability с тремя инстансами; снизить её существенно нельзя, не нарушив `RPO = 0` по платежам.

**Compute** — 18–20% инфры на старте, CUD снижает до 12–14%. Это самая «отзывчивая» статья: автоскейлер вниз ночью режет потребление, спот-инстансы для notification-консьюмера могли бы дать ещё 30–40% экономии (этой оптимизации в текущей конфигурации нет — потенциал на будущее).

**Managed Kafka** — 10% инфры. Стоимость скрытно растёт с retention'ом: при увеличении retention с 14 до 30 дней storage удваивается. Не трогаем без явной аналитической потребности.

**Application Load Balancer + Cloud CDN + egress** — 6–8% инфры. ALB фиксированно жрёт минимум за зону, CDN растёт линейно с медиа-трафиком, egress снижается с мая 2026 на 15%. Эти три статьи вместе — это «цена быть в интернете», ниже одного-двух десятков тысяч их сжать нельзя без потери multi-AZ.

В сумме на одной странице сравнения «что движет TCO» получается:

| Уровень | Категория | Доля |
|---|---|---:|
| Дороже всего | Зарплатный фонд SRE | ~70% общего TCO |
| Следующий уровень | Cloud Logging + Managed PostgreSQL | ~12% общего TCO |
| Базовый | Compute + Managed Redis + Kafka + ALB + CDN + Object Storage + egress | ~16% общего TCO |
| Минор | Container Registry + DNS + Monitoring + сертификаты | ~2% общего TCO |

Ответ на вопрос «что дороже всего и почему» — людьми, потому что у highload-системы стоимость надёжной операции структурно выше стоимости железа. Это перестаёт быть правдой только на масштабах в десятки миллионов MAU, когда compute начинает доминировать, и там уже работают другие оптимизации — собственная инфраструктура вместо managed-сервисов, спот-инстансы и переход с Go на более эффективные runtime'ы по узким местам.
