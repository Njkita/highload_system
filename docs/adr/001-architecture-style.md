# ADR-001. Архитектурный стиль: SBA + точечный EDA

- Статус: Accepted
- Дата: 2026-04-24
- Ссылки: [architecture.md §1](../architecture.md#1-архитектурный-стиль), [requirements.md](../requirements.md)

## Контекст

Проектируем сервис доставки еды под вводные из [requirements.md](../requirements.md). На выбор стиля давят четыре силы:

Разные SLA по доменам — `payment_confirmation 99.99%`, `order_create 99.95%`, `catalog 99.90%`, `notifications 99.50%` ([requirements §3.3](../requirements.md#нфт-003-availability)); в одном процессе эти классы несовместимы, отказ любого компонента обнуляет верхнее SLA для всего продукта.

Перекос read/write — пик 2849 RPS ([requirements §4.3.5](../requirements.md#435-итог-по-пику)): ~70% чтение (каталог + меню + tracking polling), лишь 48 RPS критичной записи. Scaling-policy у read и write разные; соединять их в один runtime накладно.

Противоречие «быстрый checkout» против «платежи не теряем» ([requirements §5.4](../requirements.md#54-выявленные-противоречия-и-способ-их-разрешения)) — у них разные жизненные циклы: sync create_order на секунды и долгоживущий идемпотентный payment_flow.

Ограничения команды — 3–5 инженеров, бюджет ограниченный ([requirements §5.2](../requirements.md#52-ограничения)); это исключает полноценную MSA с mesh/оркестратором саг/распределённым tracing'ом.

## Рассмотренные альтернативы

### A. Слоистый монолит

Одно приложение, слои presentation/business/data, общий PostgreSQL, один артефакт деплоя.

Дешёвый старт и отсутствие распределённых транзакций, но несовместим с разными SLA на разные домены и не даёт масштабировать read отдельно от write. Любая правка в платежах требует выкатки приложения целиком — несовместимо с 99.99% SLA на payment. Отклонён.

### B. Полноценная микросервисная архитектура

8–12 сервисов, database-per-service, Istio/Linkerd, саги через отдельный оркестратор, централизованный tracing.

Теоретически изолирует всё и даёт независимые релизы, но операционная цена выходит за бюджет: mesh и tracing сами по себе требуют выделенного инженера. Сетевые hop'ы съедают latency-бюджет checkout'а (~5–15 мс за каждый внутренний HTTP-вызов при TLS + сериализации). Саги с компенсациями — избыточный инструмент для 24 write-RPS: то, что в SBA делается одной SQL-транзакцией (create_order + outbox), в MSA превращается в распределённый протокол. Отклонён.

### C. Space-Based Architecture

In-memory data grid (Hazelcast/Ignite) + асинхронная персистентность. Сильно при непредсказуемых спайках записи.

У нас противоположный профиль — read-heavy с предсказуемыми обеденными пиками. Eventual consistency как базовое допущение несовместима с `durability RPO = 0` для платежей. Коммерческий стек grid-систем дорог, экспертиза редкая. Отклонён.

### D. SOA на ESB (SOAP/JMS)

Корпоративный ESB (Mule, IBM Integration Bus), SOAP-контракты, готовые коннекторы.

Сам ESB становится single-point-of-failure, SOAP/WSDL — не то, что ожидают мобильные клиенты, операционная стоимость на порядок выше брокера событий. Отклонён.

### E. SBA + точечный EDA — выбрана

Шесть прикладных сервисов (API Gateway, Catalog, Order, Payment, Tracking, Notification), общий PostgreSQL с логическими схемами под каждый сервис, Redis-кластер, Kafka/Redpanda, S3+CDN. Синхронный user-path короткий, REST-овый; после `COMMIT` побочные эффекты (уведомления, трекинг, обновление индекса, reconciliation) уходят через transactional outbox в Kafka.

## Решение

Принимаем SBA + точечный EDA. Границы сервисов и топология сообщений — в [architecture.md §2.2](../architecture.md#22-c4-level-2--container-diagram). Событийный слой работает только в трёх контурах: жизненный цикл заказа через `orders.events` (choreography-сага, [§3.3](../architecture.md#33-async-жизненный-цикл-заказа-через-choreography-сагу)), подтверждение платежа через `payments.events` + идемпотентный webhook + reconciliation ([§3.2](../architecture.md#32-ошибка-таймаут-psp-на-инициации-оплаты), [ADR-003](003-payment-reliability.md)), CDC `menu.updated` при миграции поиска на OpenSearch ([ADR-002](002-data-storage.md)). Всё остальное — синхронный HTTP.

## Последствия

Разные SLA теперь обслуживаются разными процессами; Payment Service изолируется сильнее остальных. Read-heavy контуры масштабируются независимо от write-path. Отказ Notification Service не блокирует checkout. Граница Payment уже проведена — при росте MAU за 30 млн первым кандидатом на выделение в отдельный runtime/БД становится именно он, без переработки контрактов.

Часть интеграций стала eventual consistent: потребитель видит `paid` через ~1–2 с после `COMMIT` за счёт outbox relay; для UX это закрыто push-уведомлением. Контракты событий — отдельная зависимость между сервисами, нужна процедура их эволюции (схемы с semver в `schemas/events/*.avro`). Без service mesh retry/timeout живут в коде — риск расхождения дефолтов, смягчается единой Go-библиотекой `internal/httpclient`.

Общий PostgreSQL — единая точка отказа, смягчается Patroni + synchronous replica и чистым разделением схем: миграция на database-per-service возможна без переписывания кода.

Отдельным ADR не выделяется выбор брокера (Kafka vs Redpanda vs RabbitMQ) — закрывается в [ADR-002](002-data-storage.md). При появлении гео-распределения придётся пересматривать `synchronous_commit=remote_apply` на `orders_db`, он не совместим с кросс-региональной репликацией.
